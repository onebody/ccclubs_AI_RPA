# AI Browser 运维指南

## 监控体系

### 指标监控

#### Prometheus 配置

Prometheus 已配置自动发现后端服务，主要监控指标包括：

- `http_requests_total`: HTTP 请求总数
- `http_request_duration_seconds`: HTTP 请求耗时
- `session_count`: 当前会话数
- `task_count`: 当前任务数

#### Grafana 仪表盘

默认创建以下仪表盘：

1. **系统概览**: CPU、内存、磁盘使用情况
2. **服务监控**: 请求量、响应时间、错误率
3. **会话监控**: 会话创建、关闭、异常数量
4. **任务监控**: 任务执行状态、成功率、耗时

### 日志管理

#### 日志收集

使用 ELK 栈收集和分析日志：

- **Elasticsearch**: 日志存储和检索
- **Logstash**: 日志收集和处理
- **Kibana**: 日志可视化

#### 日志格式

所有服务使用统一的 JSON 日志格式：

```json
{
  "timestamp": "2024-01-01T00:00:00Z",
  "level": "info",
  "service": "backend",
  "message": "Session created",
  "sessionId": "abc123",
  "tenantId": "tenant001"
}
```

## 告警策略

### CPU 告警

- **警告**: CPU 使用率 > 80% 持续 5 分钟
- **严重**: CPU 使用率 > 95% 持续 2 分钟

### 内存告警

- **警告**: 内存使用率 > 85% 持续 5 分钟
- **严重**: 内存使用率 > 95% 持续 2 分钟

### 会话告警

- **警告**: 活跃会话数 > 80% 配额
- **严重**: 活跃会话数 > 95% 配额

### 任务告警

- **警告**: 任务失败率 > 5%
- **严重**: 任务失败率 > 10%

### 服务可用性告警

- **严重**: 服务不可用持续 1 分钟

## 性能优化

### 会话管理优化

1. **会话复用**: 对于相同域名的请求，复用现有会话
2. **自动清理**: 超过最大生命周期的会话自动关闭
3. **资源限制**: 为每个会话设置 CPU 和内存限制

### 数据库优化

1. **索引优化**: 为常用查询字段创建索引
2. **连接池**: 使用 Prisma 的连接池管理数据库连接
3. **查询优化**: 避免 N+1 查询，使用批量操作

### 缓存策略

1. **Redis 缓存**: 使用 Redis 缓存频繁访问的数据
2. **会话状态**: 将会话状态存储在 Redis 中，便于横向扩展

## 安全运维

### 访问控制

1. **防火墙**: 只允许必要的端口对外暴露
2. **Ingress**: 使用 Ingress Controller 进行请求路由和安全控制
3. **TLS**: 配置 HTTPS，使用有效的 SSL 证书

### 数据安全

1. **加密存储**: 敏感数据（如密码）使用 bcrypt 加密存储
2. **传输加密**: 使用 HTTPS 加密数据传输
3. **定期备份**: 定期备份数据库，确保数据可恢复

### 安全审计

1. **审计日志**: 记录所有关键操作，便于追踪和审计
2. **权限管理**: 使用 RBAC 进行细粒度的权限控制
3. **定期检查**: 定期检查系统安全漏洞，及时修复

## 故障排查

### 会话异常

#### 现象
会话创建失败或无法正常使用

#### 排查步骤
1. 检查 Chromium 是否正常安装
2. 查看后端日志：`kubectl logs -l app=ai-browser-backend`
3. 检查会话目录权限：`ls -la sessions/`
4. 检查资源限制：当前活跃会话数是否达到上限

### 任务失败

#### 现象
任务执行失败，返回错误

#### 排查步骤
1. 查看任务详情：`GET /task/:id`
2. 检查会话状态：会话是否已关闭或异常
3. 查看后端日志，定位具体错误原因
4. 检查 AI 服务是否正常运行

### 数据库连接问题

#### 现象
服务无法连接数据库

#### 排查步骤
1. 检查 PostgreSQL 服务状态：`kubectl get pods -l app=postgres`
2. 查看数据库日志：`kubectl logs <postgres-pod>`
3. 检查数据库连接字符串配置
4. 检查网络连通性：在后端容器中 ping PostgreSQL 服务

### 性能问题

#### 现象
服务响应缓慢或超时

#### 排查步骤
1. 查看 Grafana 仪表盘，分析 CPU、内存使用情况
2. 检查请求量和响应时间
3. 查看数据库慢查询日志
4. 检查网络延迟和带宽使用

## 升级流程

### 版本升级步骤

1. **备份数据**: 升级前备份数据库和配置文件
2. **测试环境验证**: 在测试环境验证新版本
3. **灰度发布**: 先升级部分节点，观察运行状态
4. **全量升级**: 确认无问题后，升级所有节点
5. **验证**: 升级完成后，进行功能验证和性能测试

### 回滚流程

1. **确认问题**: 确认新版本存在严重问题
2. **停止新版本**: 停止新版本的服务
3. **恢复旧版本**: 部署旧版本的镜像
4. **验证**: 确认服务恢复正常

## 容量规划

### 会话容量

每个会话的资源消耗：
- **CPU**: 0.5 核
- **内存**: 512MB

### 服务器配置建议

| 用户规模 | 服务器配置 | 会话数上限 |
|----------|------------|------------|
| 小型（<50） | 4核8GB | 10 |
| 中型（50-200） | 8核16GB | 30 |
| 大型（>200） | 16核32GB | 60 |

### 水平扩展

1. **后端扩展**: 通过增加 Pod 副本数扩展后端服务
2. **数据库扩展**: 使用 PostgreSQL 主从复制或分片
3. **缓存扩展**: 使用 Redis Cluster
